// The Toolkit

A whole AI stack.
On one machine.

Projects with saved chats, smart hardware-aware model selection, a curated catalog, local-first routing with a budget, scheduled tasks & automation, RAG, optional fine-tuning, a web console, and secure multi-user access — all stdlib-only.

🧭

Smart Model Selection

Reads your CPU/RAM/GPU/VRAM and picks the best model that actually fits — and the context window adapts per request to fit your VRAM. On a CPU-only or low-VRAM box it doesn't just grab the smallest model: it chooses a capable Mixture-of-Experts model whose active params are small (≤6B/token) and whose weights fit your RAM, so you get real quality and it stays responsive.

🧠

Smart Context & Compaction

The context window auto-sizes per request so the model and its memory always fit your GPU — no silent CPU-spill slowdowns from an oversized window. When a long chat nears the limit, older turns are compressed into a rolling summary (your code blocks stay verbatim) instead of silently dropped — a badge shows how full the context is, and a conversation can keep going for weeks.

🛰️

FLEET Multi-Machine Dispatch (Pro)

Point the hub at your other computers — plain Ollama workers on your VPN — and each request runs on the node where that model is already warm, then the least-busy one, with this box as automatic fallback: a dead node never fails a request. Tag an old PC background and it quietly takes compaction, scheduled prompts, and document embeddings. Whole requests to whole models — deliberately no model splitting, which gets slower on mixed home machines. Off by default; the dashboard grows a per-node Fleet panel and the chat footer shows which machine answered. New in v0.5: a worker is one wizard to set up — it installs Ollama, the models, and even Tailscale (you sign in once) — the hub finds it with scan-and-approve, and the panel monitors every node live: requests, tok/s average + peak, uptime, CPU, RAM. The recommended agent mode keeps each worker's Ollama loopback-only behind an HMAC-signed proxy, and workers need no license.

⚡

Batch Fan-Out

Hand the server a list of independent jobs in one POST /api/batch call — 20 product photos become 20 listings at once, not one by one. Each item is a full routed request (its own images, RAG collection, model, sampling), results return in input order, one failed item never kills the rest, and cloud escalations stay budget-capped per item. Works on every tier; with a Pro FLEET the batch spreads across every machine you own — a mixed text+vision batch hits each specialist node simultaneously. Honest numbers: N warm nodes ≈ N× on independent items; a single box overlaps only as far as Ollama allows.

🗂️

Drop a Whole Project In

Zip or drag a whole project — folders + files — into the Documents tab, a Project, or straight onto the chat, and it's ingested with its folder structure intact. Citations name the real path inside the project (src/utils/foo.py, not a colliding foo.py), and a generated _STRUCTURE.md layout map lets you ask "how is this organized?". Drag-and-drop, an Add folder picker, or a .zip (best for big projects); junk dirs, binaries, and zip bombs are handled. Honest limit: answers come from retrieval (RAG) over the project — ask about specific parts — not the whole repo crammed into the context window like a 200K-token cloud assistant. Every tier.

A detect → recommend → install → verify registry adds the free runtimes a server needs — Ollama, SearXNG, Node, uv — from Set up buttons in the dashboard, the first-run wizard, and the setup screen (never sudo). // Common Questions

FAQ

Do I need a GPU?

No. Smart model selection reads your hardware and picks the best model that fits. Even on a CPU-only or low-VRAM box it doesn't just grab the smallest model — it chooses a capable Mixture-of-Experts model whose active params are small (≤6B/token) so it stays responsive, sized to fit your RAM. If you have an NVIDIA, AMD, Intel, or Apple GPU, Ollama uses it automatically for faster inference.

What happens when a chat gets really long?

It doesn’t fall apart. The context window auto-sizes per request to fit your GPU (a fixed window either wastes VRAM or silently forgets), and near the limit older turns are compressed into a rolling summary by your local model — code blocks are kept verbatim — instead of being silently dropped. A badge shows how full the context is, and you can choose “stay fast” vs “expand” per conversation.

Can it run prompts automatically on a timer?

Yes — Pro adds scheduled tasks. Add a named prompt with an hourly/daily/weekly schedule and a time of day, and the console runs it on your local model. Each result is written to data/scheduled/<id>.txt (and .json) so your other apps — like a WealthBuilder market brief or a Thrift “what’s selling” summary — can read it. You can Run now, Edit, Enable/Disable, or Delete any task. The same scheduler can also auto-refresh and (where supported) auto-train your fine-tune on a daily/weekly cadence.

Can it use more than one computer?

Yes — Pro adds FLEET. Each extra machine runs a plain Ollama daemon on your VPN; the hub routes every request to the node where that model is already loaded (no model-swap stalls), old boxes take the background work — compaction, scheduled prompts, document embeddings — and if a node dies, the request just runs on the hub. It routes whole requests to whole models and deliberately never splits one model across machines (on home networks that makes every token wait on the slowest box). Since v0.5, setting a worker up is one wizard — it installs Ollama, the models, and even Tailscale (you sign in once) — the hub finds it with scan-and-approve, the Fleet panel shows live health and tok/s per machine, and the recommended agent mode keeps the worker's no-login Ollama loopback-only behind a signed proxy. Workers need no license of their own.

My app has 20 jobs — one call or twenty?

One. POST /api/batch takes a list of independent items — each with its own prompt, images, RAG collection, even its own model — and runs them concurrently, spreading across every reachable FLEET node when you have one (a mixed text+vision batch hits each specialist machine simultaneously). You poll one id; results come back in input order, a failed item never kills the rest, and cloud escalations stay budget-capped per item. On a single box the overlap is modest (Ollama's parallel slots) — the endpoint works on every tier, and the fleet is what multiplies it.

Can I feed it a whole code project?

Yes — zip the project (or use Add folder / drag-and-drop) and ingest it, or drop the .zip straight onto the chat. It expands with its folder structure intact: citations name the real path (src/utils/foo.py), and a generated _STRUCTURE.md map lets you ask how it's organized. Junk dirs (.git, node_modules), binaries, and zip bombs are skipped automatically. Honest caveat: it answers by retrieval over the project, so ask about specific parts — a local model can't hold an entire repo in context the way a 200K-token cloud assistant can.

Is it really free to run?

Local inference costs nothing per call — you already own the hardware. The only paid path is the optional cloud fallback, which uses your own API key and stays inside the daily/monthly USD cap you set.

What does "stdlib-only" mean for me?

The app itself has zero third-party Python dependencies. It compiles to one binary with no pip, no virtualenv, no DLL hell. Inference is handled by Ollama, which the wizard installs for you.

How do my other apps use it?

Create an API key in the console, point the app's base URL at http://<host>:8765/v1, and send the key. Most OpenAI client code works unchanged. A zero-dependency Python client is included.

Can it use my files, the web, or other tools — without the terminal?

Yes. Your local model can call tools — files, web search, git, fetch — through Model Context Protocol servers, and the console installs and wires them up for you in one click; no terminal. A detect → recommend → install → verify registry sets up the free runtimes a server needs (Ollama, SearXNG, Node, uv) from a Set up button — user-level installs, never sudo. Local stdio servers keep everything on your machine; remote Streamable http servers are opt-in. Every tool call asks you to Continue or Cancel first, and over /v1 your apps opt in to tool use. Tool use unlocks at Starter; the config UI shows on every tier and installing the free runtimes is always free.

Can I put a chat box on my own website?

Yes. Embed a one-line <script src=".../widget.js"> and a floating chat panel appears, wired to your hub's /v1. It's style-isolated (shadow DOM) and configurable; a demo page is included. Use a budgeted, IP-allow-listed key for anything public.

Is anything sent to you or the cloud?

No telemetry, no phone-home. Local prompts stay on your machine. The cloud fallback only fires when you've configured a key and a hard task crosses your threshold — and you can switch it off entirely with local_only.

Can friends and apps reach it safely?

Pro unlocks secure remote multi-user. The recommended setup is a VPN (Tailscale/WireGuard/OpenVPN) with AJJ³ Brain bound to the VPN interface — nothing is port-forwarded to the open internet. A TLS port-forward path is documented as an alternative.
